# TH2W / TH2.VIP 在线游戏运营系统项目总览

## 1. 项目识别

### 项目名称

已确认事实：

- 仓库目录名是 `javaxiangmu`，但代码结构不是 Java 项目。
- `composer.json` 仍保留 Laravel 官方骨架名 `laravel/laravel`，不能作为真实业务名称。
- 前台生产入口使用 `TH2.VIP` 作为站点品牌，项目内文档使用 `TH2W` 描述活动系统。

因此，本套文档暂以 **TH2W / TH2.VIP 在线游戏运营系统** 作为业务名称。

### 项目类型

这是一个 **Laravel 6 单体应用**，同时承载：

- 面向玩家的桌面端和手机端页面
- 面向 APP / H5 的 JSON API
- 面向代理的代理中心
- 面向运营人员的 Dcat Admin 后台
- 第三方游戏平台、支付渠道、推广渠道、活动系统和运营审计能力

虽然仓库名含有 `java`，但技术栈、入口文件、依赖和目录结构均表明它是 PHP / Laravel 项目。

## 2. 项目用途

系统的核心用途是支撑一个在线游戏业务平台。代码中可确认的主要业务能力包括：

- 用户注册、登录、资料维护、API token 鉴权
- 会员余额、充值、提现、银行卡 / USDT / EBPay 等收款或提款资料
- 第三方游戏列表、游戏入口、游戏余额转入转出、投注记录、输赢记录
- WXGame 回调模式下的玩家校验、余额查询、下注、派奖、退款和状态查询
- 活动中心、活动分类、活动详情、首页弹窗、活动申请、曝光记录
- 代理团队、下级会员、代理报表、团队充值、佣金和返水
- 工单、公告、站内信、客服入口
- Dcat 后台资源管理，包括会员、充值、提现、游戏、支付、活动、代理、文章、日志等
- 新增的 TCG 风格运营后台入口，包括平台设置、推广渠道、KYC、游戏管理、运营记录和业务风控表

合理推断：

- 该系统不是单一游戏实现，而是一个聚合多个第三方游戏平台和支付渠道的运营平台。
- 近期开发重点集中在“仿 TCG 后台运营能力”“活动系统”和“前台静态入口改造”。

## 3. 技术栈概览

### 后端

- PHP：`composer.json` 要求 `^7.2`；项目文档中部分计划以 PHP 7.3 作为生产兼容环境。
- 框架：Laravel Framework `^6.0`
- 后台框架：Dcat Admin `2.*`
- 数据库：Laravel 默认连接为 MySQL，使用 `utf8mb4` 字符集。
- 队列：默认 `sync`，配置中保留 database、Redis、SQS 等 Laravel 标准队列连接。
- 定时任务：Laravel Scheduler，包含抓取游戏记录、代理返佣和多类运营审计任务。
- 测试：PHPUnit 8，仓库中已有大量单元 / 源码结构测试。

### 前端

- 桌面端入口：`public/index.html`
- 手机端入口：`public/new-h5/index.html`
- 前端实现方式：静态 HTML + CSS + 原生 JavaScript；管理后台页面大量使用 Blade、jQuery 和 Dcat Admin。
- 没有看到现代前端包管理和构建配置，因此前台更接近“静态入口 + Laravel API”模式。

### 第三方 / 外部系统

代码中可确认的外部依赖包括：

- 第三方游戏聚合网关：`TgService` 通过系统配置读取 `game_api`、`merchant_account`、`api_secret`。
- WXGame：可通过系统配置启用，支持独立 API 域名、AccessKey、回调签名、币种和 SSL 校验开关。
- 支付渠道：CGPay、ZGPay、银行卡、支付宝、微信、USDT、EBPay 等。
- Stream Chat：数据库配置文档显示系统支持 Stream Chat API Key、Secret、开关和历史消息数量配置。
- 推广渠道 / 像素：TCG shell 中包含 Facebook、TikTok、Kwai、GA4、GTM、Adjust、AppsFlyer、Voluum、Keitaro 等投放和回传配置语义。

## 4. 高层架构

系统是一个典型 Laravel 单体，但内部已经分出几类运行面：

1. **HTTP 路由层**
   - API 路由统一挂载在 `/api` 前缀下。
   - Web 路由根据域名和路径区分代理中心、手机端、桌面端、会员中心和支付回调。
   - Dcat Admin 路由使用后台配置的 prefix，并在 `app/Admin/routes.php` 中集中声明。

2. **控制器层**
   - 玩家 API 主要集中在 API 认证、首页 / 游戏 / 工单控制器、支付控制器、活动控制器。
   - 会员 Web 逻辑集中在 Member 控制器。
   - 代理中心集中在 Agent 控制器。
   - 后台 Dcat 控制器既有传统资源控制器，也有新建的自定义运营页面控制器。

3. **服务层**
   - `TgService` 封装第三方游戏网关与 WXGame 请求。
   - `SafeGameTransferService` 封装主余额与游戏平台余额之间的安全转账。
   - `PromotionService` 封装活动可见性、渠道过滤、日期过滤、排序和弹窗选择。
   - `TcgBusinessOperationService` 封装活动黑名单、活动券、游戏限制和玩家限额。
   - 后台的 `GameManagementService`、`PlatformOperationsService`、`PlatformSettingsService`、`PromotionChannelService` 封装 TCG 风格运营页面的页面契约和输入过滤。

4. **数据层**
   - 传统业务表：用户、充值、提现、银行卡、活动、游戏列表、游戏记录、转账记录、代理结算、站内信等。
   - 配置表：`system_config` 作为运行时 key-value 配置中心。
   - Dcat 后台表：管理员、角色、权限、菜单、后台设置。
   - 新增运营表：平台设置、推广渠道、游戏管理、KYC、TCG 运营记录、活动黑名单、活动券、玩家限红、用户游戏限制等。

5. **前台静态入口**
   - 桌面和手机入口直接加载共享 CSS / JS。
   - 前端脚本调用 `/api/*` 接口完成登录、会员信息、游戏启动、充值提现、活动和客服工单等流程。

## 5. 主要模块

### 用户与认证

系统同时存在 `App\User` 和 `App\Models\User` / `App\Models\Users`。前者继承 Laravel Authenticatable，承担会话认证和部分业务统计；后两者是直接面向 `users` 表的 Eloquent 模型。

API 登录后使用 `users.api_token` 作为 Bearer token。`api_auth` 中间件会根据 token 查找用户，并拒绝状态关闭、删除或黑名单用户。

### 钱包、充值、提现与转账

钱包核心是 `users.balance`。充值进入 `recharge` 表，提现进入 `withdraws` 表，余额与游戏平台之间的资金流动进入 `transfer_logs`。

重要设计是 `SafeGameTransferService`：

- 转入游戏前先锁定用户、扣减主余额、创建 pending 转账记录。
- 第三方游戏网关失败时回补余额并标记失败。
- 第三方成功但本地后处理失败时标记为“外部成功、本地待恢复”。
- 转出游戏时先创建 pending 记录，外部成功后再增加主余额。

这说明项目已经针对资金一致性做了专门加固，而不是简单调用第三方接口后直接改余额。

### 游戏与第三方平台

游戏目录主要来自 `game_lists`，平台启停来自 `apis` 和游戏自身状态字段。游戏入口会校验：

- 平台是否启用
- 游戏是否公开、PC / APP 是否可见
- 用户是否存在并可用
- 是否命中用户游戏限制
- 是否启用 WXGame 模式

传统模式通过第三方聚合网关注册、登录、查询余额、转入、转出和拉取记录。WXGame 模式则走玩家 token、回调验签、下注 / 派奖 / 退款幂等处理。

### 活动与推广

活动系统基于原有 `activities`、`activity_types`、`activity_apply` 扩展，新增排序、有效期、弹窗、桌面 / 手机图片、详情图、行动链接、按钮文案、是否需要登录、曝光记录等能力。

前台 `/activity`、`/activities`、`/promotions` 兼容同一个活动中心。活动详情支持直达 URL，首页弹窗支持频率控制。活动申请会检查登录状态、活动可见性、是否可申请、黑名单和活动券。

### 代理体系

代理系统基于用户上下级关系。`User::getChild()` 根据系统配置中的结算层级递归获取下级。代理 API 和页面提供团队报表、下级列表、业绩、返点、邀请链接、团队充值等功能。

团队充值实现了幂等控制：调用方提供 `client_order_no`，系统生成代理充值订单号，事务中同时扣减代理余额、增加下级余额，并写入充值记录和两条转账流水。

### 后台运营与 TCG Shell

后台分为两类：

- 传统 Dcat 资源页：用户、充值、提现、活动、游戏列表、支付配置、代理、文章、日志等。
- TCG 风格运营页：系统用户设置、平台设置、KYC、推广渠道、游戏管理、平台资金、风险控管、玩家等级、活动红利等。

TCG shell 提供大量菜单和页面骨架；其中部分页面已经接入真实服务和数据库，例如平台运营、平台配置、游戏管理、推广渠道、KYC 和运营记录。还有部分页面仍是基础操作区或对标页面语义，真实业务逻辑需要继续接入。

### 审计与质量保障

代码中有两类审计：

- 业务操作日志：如用户操作、代理操作、后台表单变化。
- 命令型运营审计：如游戏、前台、API、钱包、后台审计命令，定时输出日志和报告。

测试重点不是传统 HTTP 集成测试，而是大量“源码结构测试”和服务行为测试，用来约束后台页面契约、路由、迁移字段、前端入口和关键服务行为。

## 6. 关键设计特征

### 以 `system_config` 作为运行时配置中心

许多业务开关和第三方参数不是写死在配置文件，而是存入 `system_config`：

- 游戏网关地址和密钥
- WXGame 配置
- 充值 / 提现上下限
- 提现时间窗口
- Stream Chat 配置
- 平台设置、下载链接、前台样式、APP 打包参数

这种方式适合运营后台动态调整，但也意味着配置变更需要更强审计和回滚能力。

### 兼容旧接口，同时新增语义更清晰的新接口

活动系统保留旧的 POST 活动接口，同时新增 `/api/promotions/*` 只读接口。工单也保留多套命名兼容入口。游戏回调同时支持 `/api/wxgame/*`、`/api/notify/*` 和 Web 下的 `/notify/*` 兼容路径。

这体现了向新前台 / 新后台迁移时尽量不破坏旧客户端的取舍。

### 资金链路开始从“直接写余额”转向“事务 + pending 状态”

旧转账逻辑仍保留在控制器中，但在新入口处已经先返回安全转账服务结果，旧逻辑实际不可达。资金链路正在从控制器内的直接改余额，迁移到服务内的事务、锁、外部状态、恢复状态和对账备注。

### 后台页面采用“页面契约”驱动

新后台服务将每个 TCG 页面抽象为页面代码、标题、存储表、筛选项、列、字段、动作权限和日期字段。这样可以用同一套 Blade 视图和控制器处理多个运营页面。

优势是能快速补齐大量后台页面；风险是页面契约会变成新的“配置型复杂度”，需要测试严格约束。

## 8. 可见风险

### 控制器过大，领域边界不够清晰

API 首页控制器、支付控制器、会员控制器、代理控制器都非常大，混合了鉴权、参数处理、业务规则、数据库写入、第三方调用和响应格式。后续维护时容易出现局部修改影响跨模块行为。

### 用户模型重复

`App\User`、`App\Models\User`、`App\Models\Users` 都指向用户概念。不同地方混用不同模型，会增加关系、鉴权、类型和命名误判风险。

### 旧逻辑残留较多

部分控制器方法在新安全服务 `return` 后仍保留大段旧实现，实际不可达。短期有助于回退参考，长期会干扰阅读和维护。

### 路由兼容导致接口面膨胀

同一业务存在多个路径、多个命名和重复路由。例如充值、银行卡、工单、通知回调等。兼容性强，但 API 语义和权限边界更难审计。

### 安全边界需要持续审计

已看到的风险点包括：

- API token 存在前端 localStorage 中。
- CORS 中间件会把请求 Origin 回写为允许来源。
- 部分老接口手动解析 Authorization token，未统一依赖 `api_auth` 中间件。
- 支付、游戏回调、资金操作涉及高风险业务，需要继续用审计命令和集成测试约束。

### 部署证据不足

仓库中没有看到 Dockerfile、docker-compose、GitHub Actions 或其他 CI/CD 配置。部署方式、环境变量清单、进程管理和回滚策略目前证据不足，后续若生成部署文档需要谨慎标注。
